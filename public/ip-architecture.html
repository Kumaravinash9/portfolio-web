Part 1The problem & the setup

1Introduction

This is the architecture of our Document Intelligence Ingestion Pipeline for EPC engineering documents. Every design decision below comes directly from the problem it solves, so we start with the problem.

In EPC projects, engineers work with large governing documents: SOWs, PMS, HMBs, design philosophies, HAZOP reports, SIL reports. Each one can be 200 to 800 pages long.

Now imagine a piping engineer wants to know: "Is a 4-inch Gate Valve allowed in Pipe Class A1A?" Or a process engineer wants: "What is the pressure, temperature and composition of Stream 105?"

Today the workflow is completely manual. The engineer opens a PDF, searches with Ctrl+F, checks different revisions, reads multiple tables, verifies footnotes, and finally gets an answer. Sometimes it takes 15 minutes, sometimes an hour, and sometimes they still end up using an outdated revision.

Our objective is simple: engineers should ask a question in plain language and get an accurate answer, with complete traceability back to the source document.

The goal: replace "open the PDF and Ctrl+F for an hour" with "ask a question, get a cited, current answer."

2Why not just use normal RAG?

The obvious question is: why not simply use a traditional RAG pipeline? Because engineering documents are fundamentally different from normal documents. A normal RAG pipeline looks like this.

This approach often struggles with engineering documents, because tables, notes, and clauses can get split across chunks. Take a PMS table:

Pipe ClassValveTemperaturePressureNotes
A1AGate Valve300°C150#See Note 4

Now imagine the chunker splits this one table into two chunks:

Chunk One
Pipe Class · Valve · Temperature
✂
Chunk Two
Pressure · Notes

Suppose Note 4 says "Gate Valve is NOT permitted above 250°C." But that note went into the other chunk. When the LLM retrieves only Chunk One, it confidently tells the engineer:

Wrong answer: "Yes, Gate Valve is allowed." The exception was in a different chunk. The problem was not the LLM. The problem was our ingestion strategy.

Same thing with an HMB. If a stream row gets divided, the Pressure lands in one chunk and the Flow in another, so Stream 105 no longer exists as a complete record. Again, a wrong answer.

So our philosophy is: engineering documents should never be treated as plain text. They should be treated as structured engineering knowledge.

3Our core philosophy

Spend compute once during ingestion, save compute during every query.

Instead of asking the LLM to understand the document every single time somebody asks a question, we spend the heavy compute once during ingestion, understanding the document completely. For a typical digital document that is on the order of tens of minutes; a large, fully scanned document can take longer. Real timings come from the pilot, and because the work runs in the background, once per document revision, nobody is ever waiting on it.

Once that is done, every future query becomes faster, more accurate, more explainable, and much cheaper.

6Stage 0 — Intake & Registration

The first stage is Intake and Registration. One important thing to notice: the very first thing we do is not OCR. We don't parse the document, we don't generate embeddings. We first register the document inside the platform.

Why? Imagine a customer accidentally uploads the same SOW five times. Without registration, we would process it five times. That is wasted compute.

Every document gets a unique identity

Document IDProject IDRevisionDocument Number Upload TimestampSourceUploader

Hashing (a fingerprint for the file)

We generate a SHA-256 hash of the document. Think of it as a fingerprint. Suppose today's file produces a hash like A1B9C8F7... Tomorrow the client uploads the same file, we hash it again, and if the two hashes are identical we immediately know this document is already processed. So we reuse the existing results. No duplicate computation.

Metadata (these become search filters later)

Client NameProject NameRevision Number Issue DateDisciplineDocument Number
Because of metadata, instead of "Show Stream 105" an engineer can ask "Show Stream 105, only from Revision C" or "only from Project ABC." Without metadata, that is impossible.

MDR reconciliation Later phase

If the project already has an MDR (the Master Document Register), we can reconcile expected vs received vs successfully ingested, so project managers see which documents are still missing. This is a dashboard feature layered on later, not part of the core ingestion path, so it never blocks a single document from being processed.

Expected documents
↓
Received documents
↓
Successfully ingested
Stage 0 is deliberately thin: hash, identity, metadata. Just enough to prepare the document before any AI processing begins.

7Stage 1 — Document Classification

Now classification. The first question we answer is: what exactly is this document? We keep this cheap. The default comes from the upload form and filename, confirmed by one small LLM call on the first few pages (sent as page images, so scanned documents classify without waiting for OCR), with a manual override the user can always set. No heavy classification subsystem.

SOWPMSHMBDesign Philosophy HAZOPSILMDR

Why classification matters: each type needs a different schema

If it's an SOWI'm not looking for stream values. I want obligations: "Contractor shall submit...", "Vendor shall provide...". Schema: Responsible Party, Deliverable, Milestone, Clause.
If it's a PMSNow I need Pipe Classes, Materials, Pressure Ratings, Temperature Limits, Allowed Components. A completely different schema.
If it's an HMBEverything changes again: Stream Numbers, Pressure, Temperature, Flow, Composition. Different prompt, different schema, different registry.

So classification decides which pipeline should process the document.

Edge cases: multi-volume and mixed files Manual override in v1

Two harder cases exist. Multi-volume: one SOW arrives as Volume I, II, III plus appendices that should be grouped into one logical document. Mixed file: a single PDF whose first 150 pages are SOW and next 200 are PMS. Auto-detecting these reliably is genuinely hard and genuinely rare, so in v1 we handle them through the manual override: the reviewer groups the volumes or marks the split. This is a designed step in the upload flow, not an afterthought: a multi-file upload asks "are these volumes of one document?" right at upload, so grouping is one click while the uploader still has the context. The common case (one file, one type) stays fully automatic and loses nothing, and auto-detection (filename patterns, MDR entries, first-page titles) can be layered in later without changing anything downstream.

Volume IVolume IIVolume IIIAppendices→ grouped by reviewer
After Stage 1, the system knows what kind of engineering knowledge to extract.

8Stage 2 — Parsing & Layout Understanding

This is one of the most challenging stages: understanding document layout. The biggest mistake people make is assuming every PDF is just text. Engineering PDFs are far more complicated.

Some are digital, some are scanned. Some have landscape tables, engineering drawings, handwritten notes, merged cells, rotated pages, or no text layer at all. So instead of one parser, our parsing pipeline is adaptive.

PDF
↓
Layout Parser
↓
OCR Engine (if required)
↓
Vision Parser
↓
Structured layout
The names above describe what each step does, not which library does it. Current candidates — Layout: Docling · OCR: Tesseract · Vision: GPT-4o Vision. Swapping a candidate never changes the architecture.

Layout Parser — understands structure, not just text Planned Docling

The first component is the layout parser (current candidate: Docling). It does not simply extract text, it understands document structure. For every page it identifies:

HeadingsSectionsParagraphsTables ImagesHeadersFootersReading order

So instead of one giant block of text, we now have a structured representation. Take a PMS table:

Pipe ClassValveTemperatureNotes
A1AGate Valve300°CSee Note 4
If we only extract text, the link between the columns disappears. The layout parser preserves it: it knows this value belongs to this column, this note belongs to this row.

OCR Engine — for scanned pages Planned Tesseract

If the layout parser detects a page has no text layer, we switch to the OCR engine automatically. Our current candidate is Tesseract, because many EPC and government documents are scanned and sometimes contain both English and Hindi. It converts the image into machine-readable text.

Vision Parser — for tables trapped in images Planned GPT-4o Vision

But OCR alone is not enough. In an HMB, stream tables are sometimes embedded inside diagrams or complex scanned images. The OCR engine can read the words but cannot understand the table structure. So we use a vision parser: a vision-language model. It understands visual relationships: rows, columns, merged cells, branch matrices, engineering diagrams, table hierarchies. It looks at the page almost the way a human engineer would.

How we de-risk this stage: the layout parser, OCR engine, and vision parser are where most of the technical risk in this design lives, so they get proven first, not last. Week one of the build is a parsing spike: run the candidate parsers on real pages from our own corpus (SOW, PMS, HMB; scanned and digital), score table-cell accuracy against hand-checked pages, and pick winners from the data. Nothing downstream changes based on which parser wins, only the internals of this stage.

Part 2The intelligence layer (the key differentiator)

9Stage 3 — Page-Based Chunking with Metadata

So far we have only understood the document. No embeddings, no vector database, and no LLM calls yet beyond Stage 1's small classification check. Now we cut the document into chunks, and here we deliberately keep things simple.

Instead of complex clause-level segmentation, we chunk by page: one page = one chunk. Pages are the natural unit engineers already think in ("see page 87"), the logic is predictable, fast to build, and easy to debug and maintain.

Traditional RAG splits blindly
500 tokens
↓
500 tokens
↓
500 tokens
We chunk by page
Page 86 chunk
↓
Page 87 chunk
↓
Table chunkmerged if it spans pages

The one exception rule: a table is never cut

Page boundaries have one danger: a PMS or HMB table that continues onto the next page. So we keep exactly one exception. A table detected in Stage 2 always becomes its own chunk, and if it spans multiple pages, those parts are merged into a single chunk together with its notes. Everything else follows the page boundary.

This keeps the "Note 4" problem solved: the note that says "Gate Valve is NOT allowed above 250°C" stays in the same chunk as its table, so the LLM can never see the row without its exception.

Every page chunk carries rich metadata

Page IndexDocument IDDocument TypeRevision ProjectSectionChunk Type (text / table)Bounding Box

The section comes free from Stage 2, which already knows which heading governs each page. Before indexing, we prepend a short context line built from this metadata to every chunk, so its embedding and keyword entry know exactly where the page lives:

[PMS.pdf | Rev C | Section 4.2 · Pipe Class A1A | p.87] ...page content follows...
Why this works: the metadata gives us revision-aware filtering ("only Rev C"), page-level citations ("Section 4.2, page 87"), and the context line makes every page findable even when the page itself never repeats the class name or document type.
The honest trade: citations are page-level instead of clause-level. Typed Extraction (Stage 4) is unaffected, it reads Stage 2's structured output directly, so the registries stay exactly as designed.

10Stage 4 — Typed Structured Extraction

This is where the real intelligence begins. So far we only organized the document. Now we convert engineering documents into structured knowledge. Our philosophy is simple: different document, different extraction schema. We never use one generic prompt for everything.

Example 1 — SOW → Obligation Registry

The AI reads: "The Contractor shall submit Piping GA Drawings before Mechanical Completion." Instead of storing the paragraph, we extract structured fields:

{ "Responsible Party": "Contractor", "Deliverable": "Piping GA Drawings", "Milestone": "Mechanical Completion", "Source": "Clause 6.4" }
Later, if someone asks "show me every contractor obligation," we don't search PDFs. We query the Obligation Registry directly.

Example 2 — PMS → Material Registry

{ "Pipe Class": "A1A", "Valve": "Gate Valve", "Temperature": "300°C", "Pressure": "150#" }

Now if an engineer asks "Can I use a Gate Valve in A1A?" that is a structured lookup, so it goes straight to the Material Registry, no vector search needed. Very fast.

Example 3 — HMB → Stream Registry

{ "Stream": "105", "Pressure": "25", "Temperature": "320", "Flow": "120" }

Examples 4, 5, 6 — Philosophy, HAZOP, SIL

Design Philosophy → Rule Registry"Double Block and Bleed shall be provided for toxic service" becomes: Condition, Rule, Scope, Exceptions.
HAZOP → Action Registry"Install Pressure Alarm, Responsible: Instrumentation, Status: Open" becomes a structured action row.
SIL → SIF Registry"Pump Shutdown, SIL 2, PFD, IPL Credits" becomes a structured safety-function row.

So by the end of Stage 4, our PDFs have become structured databases:

Obligation RegistryMaterial RegistryStream Registry Rule RegistryHAZOP RegistrySIF Registry

Implementation: six registries, one physical table

Six registries does not mean six tables, six write paths, and six migration histories. Everything around the typed fields is identical for every record: provenance, confidence, HITL status, revision link, pending/published flag. Only the payload differs, and that is enforced by a per-type schema (Pydantic) at write time, not by table DDL. So all six registries live in one physical table, exposed as six SQL views:

extracted_records -- one physical table record_type -- 'obligation' | 'material' | 'stream' | 'rule' | 'hazop' | 'sif' payload JSONB -- typed fields, validated against the record_type schema before write doc_id · revision · status · confidence · page · bbox · original + canonical value obligation_registry, stream_registry, ... -- SQL views over this table; -- hot fields (stream_id, pipe_class, party) -- get generated columns + indexes
Why this matters: one write path, one HITL review UI, one provenance model, one migration history. Adding registry #7 later is a new prompt plus a new schema, zero database changes. Every registry mentioned in this document is one of these views, and queries against them look and perform exactly like dedicated tables.
We don't build all six at once. Each extractor is just one prompt plus one schema, added phase by phase: Obligation first, then Material and Stream, then the rest. Extraction quality per registry is identical, there is simply less to build and maintain at any one time. Each new extractor launches with its own small golden set (Stage 7) and full HITL coverage, so early mistakes are caught by process, not by users. Document types whose extractor hasn't shipped yet are still fully ingested through the search branch (page chunks, semantic + keyword search), so they are queryable from day one, just without the typed fast path.
We route each question to the cheapest path that can answer it correctly: structured questions are answered from the registries with SQL, while open-ended questions use hybrid retrieval over the indexed page chunks. So "pressure of Stream 105" is a millisecond registry lookup, and "what does the isolation philosophy say about toxic service" uses semantic + keyword search. We don't force every question through embeddings.

11Unit Normalization

Another EPC challenge: different clients use different units, like bar, kg/cm², psi, MPa. If we store them raw, search becomes difficult. So we normalize, while keeping the original. This runs right after extraction and before validation, so every downstream check, and every HITL reviewer, sees the original and the canonical value together.

Original: 25.5 kg/cm²
↓ normalize (1 kg/cm² = 0.980665 bar)
Stored: 25.0 bar
+ keep original verbatim
This gives us consistent querying without losing the original engineering evidence. The conversion is a deterministic lookup table, not an LLM, and Stage 4.5 re-runs it as one of its checks.

12Stage 4.5 — Audit & Validation Pass

Now a very important reliability step. We never blindly trust a single LLM.

Checks pass

↓

↓ pass

Checks fail or score low
Deterministic checks
↓ fail / low score
LLM audit
↓ still unresolved

We validate cheaply first. Before spending any second LLM call, we run deterministic checks: is the schema complete, is the unit valid, does the canonical conversion recompute exactly, and most importantly, does the original extracted value appear verbatim in the parsed source text? If a record stores original 25.5 kg/cm² and canonical 25.0 bar, we string-match "25.5" back to the actual table cell and re-run the conversion. A string match cannot hallucinate. Only records that fail these checks or score low get the expensive second LLM audit, then HITL if still unresolved. Same guarantee, a fraction of the LLM calls.

Confidence & Human-in-the-Loop

Every extracted record gets a confidence score, for example 0.98, 0.91, or 0.42. Anything below the threshold goes to HITL, Human in the Loop. The reviewer sees the original PDF, the extracted JSON, and a correction panel. If a correction is needed, the human edits it and the database updates.

0.98 ✓0.910.42 → HITL
In engineering, being confidently wrong is worse than saying "I don't know." One incorrect pressure value can cause a wrong equipment selection. Our rule: never silently wrong. If confidence is low, ask a human.

13Stage 5 — Revision Handling

Now something unique to EPC projects: engineering documents constantly evolve. Suppose Revision A says Pressure = 20 bar, and later Revision B changes it to 25 bar. Traditional RAG stores both, so retrieval sometimes returns the old chunk, sometimes the new one. Very inconsistent.

Our solution: a supersedes link, not a graph engine

Revision A
↓ superseded by
Revision B
↓ superseded by
Revision C (current)

We don't need a dedicated graph database for this. Every document carries one supersedes reference to the revision it replaces, and every registry record carries its revision. When an engineer asks "pressure of Stream 105," the query simply filters to the latest revision. Revision C, not A, not B. One column and one filter deliver the core guarantee: superseded content is never served as current.

Conflict detection comes for free

Suppose the PMS says 300°C and another current document says 280°C. We don't need a special ingestion-time conflict detector. Because both records are stored with citations, hybrid retrieval naturally returns both, and the answer layer presents them side by side with sources. The SQL fast path follows the same rule: if a registry query returns more than one current row for the same key, the answer layer treats that as a conflict and shows both records with their citations. The "never invent agreement" behavior falls out of the retrieval design itself.

Cross-reference links Later phase

Documents refer to other sections: "Refer Clause 5.2," "Refer Table 8." Turning these into clickable navigation is a nice UX touch, but it does not change answer quality, so we defer it. The section metadata we already store is enough to resolve them later without re-ingesting.

Our rule: never invent agreement where the documents disagree. By the end of Stage 5 we have structured, validated, revision-aware engineering knowledge, with one column instead of a graph engine.
Part 3Making knowledge searchable & the payoff

14Stage 6 — Indexing: One PostgreSQL, Five Logical Views

Simplified Earlier this was drawn as five separate stores. In practice, at our scale, all five are one PostgreSQL instance: registries as typed views over one records table, pgvector for embeddings, full-text search for keywords, provenance as columns, and one supersedes link for revisions. One database to deploy, back up, and run on-prem.

The key insight still holds: not every question should be answered with embeddings. Different questions need different access paths. But those paths don't need five different systems, they are five logical views inside a single database.

View 1 — Registries (six SQL views over one records table)

All extracted entities live here, in the single extracted_records table from Stage 4, exposed as one indexed view per registry. If an engineer asks a structured question like "what is the pressure of Stream 105?" we don't need an LLM or vector search, just SQL:

-- pressure of Stream 105 SELECT pressure FROM stream_registry WHERE stream_id = '105'; -- result: 25 bar. Milliseconds. No hallucination.
-- every contractor obligation SELECT * FROM obligation_registry WHERE party = 'Contractor';
Our rule: if structured data already exists, don't ask the LLM.

View 2 — Semantic search (pgvector, same database)

Not every question is structured. "What does the isolation philosophy say about toxic service?" is a paragraph, not a field. So we embed the page chunks into a pgvector column. Even if the document says "Double Block and Bleed," semantic search understands the concepts are related. No separate vector database to run, the vectors sit right next to the registries.

View 3 — Exact identifiers (Postgres full-text search)

Embeddings have one weakness: engineering identifiers like A1A, NACE MR0175, API 610, ASME B31.3, ISO 14692. Embedding models don't always preserve these correctly. So we also keep a lexical index using PostgreSQL full-text search (BM25-style ranking). "Show every reference to NACE MR0175" is exact matching, and lexical search beats embeddings there. Hybrid retrieval = View 2 + View 3 together.

View 4 — Provenance (columns on every row)

Engineers never blindly trust AI, they ask "where did this come from?" So every record carries its provenance as columns: Document, Revision, Page, Section, Bounding Box. When the AI answers "Pressure = 25 bar" and the engineer clicks, the system opens the exact page, because we stored the coordinates during ingestion. No separate citation service, just fields on the row.

View 5 — Revisions (one supersedes link)

Finally, the revision chain from Stage 5: a single supersedes foreign key connects Revision A → B → C. That one column is what lets every query filter to the current revision. No graph database required.

The final knowledge base — one database

PostgreSQL (single instance) │ ├─ extracted_records (one table) │ ├─ views: Obligation / Material / Stream │ └─ Rule / HAZOP / SIF ├─ pgvector column → semantic search ├─ full-text index → exact identifiers ├─ provenance columns → citations └─ supersedes link → revisions
Why one database: simpler to deploy on-prem (a hard requirement), provenance is a plain foreign-key join instead of a cross-system lookup, and there is one thing to back up and secure, not five. Split a view out into a dedicated engine later only if scale ever forces it. Everything Stage 6 writes lands with status = pending; nothing becomes queryable until Stage 7 promotes it.

15Stage 7 — Evaluation & Quality Gates

One final stage before ingestion is considered complete: evaluation. We don't want bad engineering knowledge entering production, so every ingestion run is graded against a golden dataset.

We don't hand-build huge golden sets upfront. We start with a small seed set per document type plus spot-check sampling on each run, then grow the golden set automatically from the corrections reviewers make in HITL. The evaluation gets stronger over time without a big labeling project blocking the launch.
ObligationsGolden set says the SOW has 120 obligations. Our extractor finds 118. Accuracy 98%. Good.
StreamsHMB has 300 streams. Extractor finds 295. Accuracy 98.3%.
CitationsPressure extracted correctly but linked to the wrong page? That is a failure, because engineering users need evidence.

What we evaluate

Extraction AccuracyRegistry CompletenessCitation Accuracy OCR QualityTable ParsingRevision ResolutionOverall Confidence
Publication is an explicit gate: every record is written as status = pending, and only when the metrics pass does Stage 7 flip it to published and visible to queries. If the metrics fail, nothing goes live and the document goes back for review. Never compromise engineering accuracy.

18Why this is better than traditional RAG

Traditional RAG

PDF → Chunk → Embed → LLM
  • Breaks tables
  • Breaks clauses
  • Doesn't understand revisions
  • No structured knowledge
  • Hallucination risk
  • Slow for numerical queries

Our architecture

Understand → Page chunks + metadata → Extract typed knowledge → Normalize → Validate → Revision link → Registries → One Postgres (many views)
  • Keeps tables and notes together
  • Simple, predictable page chunks
  • Revision-aware by design
  • Structured registries, queried directly
  • Validated, never silently wrong
  • Millisecond lookups for numbers
The biggest difference: most RAG systems treat documents as text. We treat documents as engineering knowledge. That single decision changes the entire pipeline.
The trade-off: this design spends more compute up front during ingestion (tens of minutes for a typical document, potentially a few hours for an 800-page fully scanned one; the pilot gives us real numbers) in exchange for much faster, more accurate, and more reliable query-time performance. The bet is that a document is ingested once per revision but queried thousands of times, so paying the cost once is worth it, and since ingestion runs in the background, no engineer ever waits on it.

19Future scope & closing

Once these registries exist, building future products becomes much easier: cross-document validation, drawing validation, datasheet validation, engineering compliance, vendor deviation analysis, even agentic engineering workflows. All of them can consume the same knowledge layer.

So this pipeline is not just preparing documents for question answering. It is building the foundation of an Engineering Knowledge Graph for the entire platform.

To conclude, our ingestion pipeline is built around one core philosophy: understand the document once, structure the knowledge once, validate it once, and reuse that knowledge everywhere.

Instead of repeatedly asking an LLM to understand the same PDF on every query, we do the heavy intelligence upfront during ingestion. The result is a system that is faster, more accurate, revision-aware, fully traceable, and significantly more reliable for engineering workflows.
